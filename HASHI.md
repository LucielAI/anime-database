## What this product is

Anime Architecture Archive (animearchive.app) maps anime universes as engineered systems — not plot summaries, structural analysis. Each universe is a JSON payload that drives a React SPA with specialized visualizations (Timeline, NodeGraph, CounterTree, AffinityMatrix). The product runs on Vercel. Data lives in `src/data/`. Everything the UI renders comes from those JSON files.

The product serves two goals simultaneously: portfolio-grade technical depth and TikTok/social shareability. Both matter equally.

## What requires founder check-in before you act

These touch the live deployment or have outsized blast radius. Flag them, explain your reasoning, wait for a go-ahead.

| Area | Why it needs sign-off |
|---|---|
| `vercel.json` — any change to routes, rewrites, or headers | Directly controls how Vercel serves the site. Wrong change = production outage or broken routing |
| `middleware.js` | Runs on every request at the edge. Bugs affect all users immediately |
| `.github/workflows/ci.yml` | Changes to CI affect the safety net for all future merges |
| Any new file in `api/` | New serverless functions are live on deployment; security and routing implications |
| `supabase/migrations/` | Database schema changes are irreversible |
| Changing how universe data loads at runtime (`src/data/index.js`) | All 30 universes depend on this; breakage is total |
| Changing how meta/OG tags are delivered | Architectural decision already made — see ADR-024 and REJECTED APPROACHES in CLAUDE.md |
| Deleting or restructuring `public/` | Static assets affect deployed URLs |

If you are unsure whether something falls in this list, it probably does. Flag it.

## Operating principles

**You are working on a live site.** Changes that merge to Main deploy to real users. Test before you push. `npm run build` must pass. `npm run test` must pass. `npm run lint` should be clean.

**Read before you change.** Never modify a file you haven't read. Never guess at what existing code does.

**Smallest correct fix.** Do not refactor things adjacent to what you were asked to fix. Do not add features that weren't requested. Scope discipline is how you stay trustworthy.

**If something looks wrong in the codebase, say so.** You are a co-founder. If you find a bug, dead code, or a bad pattern that isn't in your current task, note it — don't silently fix unrelated things, but do surface it.

**Check the rejected approaches before any SEO or rendering idea.** The static HTML pre-rendering pattern broke production twice. It is documented in `CLAUDE.md` under REJECTED APPROACHES and in ADR-024. If you ever think "what if we pre-render HTML for universe pages" — read those sections first. The answer is no.

**Your commits are part of the public record.** Write them like someone will read them in 6 months to understand what you were thinking.

## When something breaks

1. Read the error. Actually read it.
2. Check `npm run build`, `npm run test`, `npm run validate:all` to isolate where.
3. Look at recent commits — what changed just before the break?
4. Fix the root cause, not the symptom.
5. If the fix would touch anything in the "requires check-in" list above, stop and flag it.
6. Do not use `--no-verify`, `--force`, or skip CI gates to push through a broken state.

## A note on trust

You have broad latitude here because the founder trusts you to use it well. That trust is maintained by:
- Being transparent about what you're doing and why
- Flagging things you're uncertain about before acting
- Not taking shortcuts that trade short-term speed for long-term stability

That is the cost of an agent operating without enough context about what not to do. This file exists so that doesn't happen again.
